## 1. Общая схема архитектуры (Компоненты)

Архитектуру MVP можно разделить на 4 ключевых слоя:

```
[ Frontend: Web UI / Chat ] 
          │
          ▼
[ API Gateway / Auth (СберID / IAM) ]
          │
          ▼
[ Orchestration Layer: Multi-Agent System (LangChain/LangGraph) ] ─── [ Vector DB / Cache ]
          │                                        │
          ▼                                        ▼
[ LLM/AI Services (GigaChat API / On-Prem) ]  [ ETL / Document Processing Pipeline ]
                                                   │
                                                   ▼
                                      [ Corporate Cloud (СберДиск API) ]

```

## 2. Архитектура слоев и технологический стек

### А. Слой хранения и интеграции (Data & Storage Layer)

Слой отвечает за взаимодействие со структурированными и неструктурированными данными из корпоративного облака.

* **Интеграция:** Подключение к API СберДиска (или тестовому S3-совместимому хранилищу для MVP) для получения файлов, метаданных и отслеживания изменений (вебхуки или периодический поллинг).
* **База данных метаданных:** **PostgreSQL** (для хранения структуры папок, связей, метаданных, логов сессий пользователей и истории чатов).
* **Векторная база данных (Vector DB):** **Qdrant**, **Milvus** или расширение **pgvector** для PostgreSQL. Необходима для хранения эмбеддингов документов и обеспечения семантического поиска (RAG).

### Б. Слой обработки документов (ETL / Document Pipeline)

Асинхронный пайплайн для обработки «тяжелых» корпоративных файлов (PDF, DOCX, XLSX, TXT).

* **Очередь задач:** **Celery + Redis** (или RabbitMQ) для асинхронной обработки файлов без блокировки основного интерфейса.
* **Парсинг и чанкинг:** **Unstructured** или **LlamaIndex** для извлечения текста и деления его на оптимальные чанки (с учетом сохранения контекста таблиц и списков).
* **Детекция дубликатов:** Использование хэширования (SimHash/MinHash) для текстовых дубликатов на этапе ETL.

### В. Слой ИИ и оркестрации (AI & Multi-Agent Layer)

Сердце системы, реализующее концепцию мультиагентности, описанную в кейсе.

* **Оркестратор агентов:** **LangGraph** (идеален для циклических графов и четко контролируемых агентов) или **CrewAI / AutoGen**.
* **Разделение агентов на роли (MVP-набор):**
1. **Router Agent:** Анализирует промпт пользователя (например: «Найди дубликаты отчетов за май» vs «Сделай саммари проекта Х») и направляет задачу специализированному агенту.
2. **Search & RAG Agent:** Извлекает контекст из векторной БД, формирует финальный промпт для LLM.
3. **Analytics & Tagging Agent:** Отвечает за генерацию тегов, построение ментальных карт (граф-схем взаимосвязей документов) и выявление дубликатов.
4. **Summarizer Agent:** Оптимизирован под сжатие больших объемов текста.


* **LLM (Языковая модель):** Интеграция с **GigaChat API** (как базовое решение для Сбера) или локально развернутая open-source модель (например, Llama-3 / Mistral) через **vLLM / Ollama** для соблюдения контура безопасности.

### Г. Backend API и Безопасность

* **Фреймворк:** **FastAPI** (Python). Обеспечивает высокую скорость, асинхронность и автогенерацию OpenAPI-документации.
* **Безопасность (Enterprise Ready):**
* Интеграция с корпоративным IAM (OAuth2 / Active Directory / СберID).
* **ACL (Access Control List):** Агент должен иметь доступ *только* к тем файлам, к которым у текущего пользователя есть права в СберДиске. Токены доступа транслируются в запросы к API хранилища.



### Д. Frontend (Интерфейс пользователя)

* **Фреймворк:** **React.js** (с TypeScript) + Tailwind CSS (или корпоративный дизайн-кит Сбера).
* **Компоненты UI:**
* Диалоговое окно (Chat UI) со стримингом ответов (Server-Sent Events / WebSockets).
* Файловый менеджер / Дашборд, отображающий «карту знаний» (библиотеки визуализации графов вроде **React Flow** или **vis.js** для отображения связей между документами).

## 3. Ключевые архитектурные паттерны для этого MVP

1. **Advanced RAG (Retrieval-Augmented Generation):** Не просто базовый поиск по сходству, а гибридный поиск (Keyword BM25 + Vector Semantic Search) с последующим переранжированием (Reranking) через Cross-Encoder. Это критично для точных ответов на вопросы «Быстрее, чем искать прямой поиск документа», указанные в целях кейса.
2. **State Management в LangGraph:** Состояние (State) агента должно сохраняться в PostgreSQL, чтобы пользователь мог прервать сессию, вернуться к ней или увидеть историю рассуждений агента (Chain-of-Thought).
3. **Idempotency & Event-Driven Processing:** СберДиск генерирует события изменения файлов $\rightarrow$ ETL-пайплайн обновляет векторные индексы на лету, чтобы AI-агент всегда работал с актуальной версией документов.

Такой стек и структура позволят собрать рабочий прототип за несколько недель, при этом архитектура легко масштабируется от MVP до полноценного пилота внутри инфраструктуры банка.
